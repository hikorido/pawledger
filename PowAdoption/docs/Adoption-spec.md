# PawLedger项目-宠物领养模块 产品需求文档（PRD）
## 一、文档概述
### 1.1 文档目的
明确宠物领养模块的产品需求、功能规则、业务流程，为开发、测试、路演交付提供统一的标准依据，同时明确本模块与项目现有筹款模块的边界与对接规则，确保模块可独立运行、也可无缝融入项目整体架构。
### 1.2 适用范围
本文档仅针对宠物领养模块单独说明，适用于该模块的合约开发、前端开发、测试验收、路演讲解全流程，不涉及项目其他模块的需求变更。
### 1.3 术语说明
| 术语 | 大白话解释 |
|------|------------|
| 链上 | 指Avalanche区块链网络，所有数据写入后不可篡改、永久留存、可公开查询 |
| 哈希指纹 | 通过SHA256加密算法生成的唯一字符串，原文任意修改都会导致哈希值完全变化，不可逆，无法通过哈希值反推原文 |
| 救助人 | 发布宠物领养信息的用户，一般为流浪动物救助者/救助机构 |
| 领养人 | 申请领养宠物的用户 |
| 合约 | 部署在链上的代码程序，所有规则、操作逻辑均写死在合约内，执行结果不可篡改 |

## 二、需求背景
### 2.1 行业核心痛点
当前国内流浪动物领养场景，90%以上依赖中心化平台、社群、线下机构完成，存在无法解决的行业顽疾：
1.  中心化平台的领养人信息存储在平台服务器，可被随意删除、篡改，一旦出现领养人弃养、虐待动物的情况，极易出现信息灭失、领养人抵赖失联的问题，救助人维权无门；
2.  虐宠团体可通过虚假信息在不同平台反复领养动物作案，领养信息不互通，作恶成本极低，溯源难度极大；
3.  领养完成后，救助人无法对领养人形成有效约束，没有可信的凭证证明领养行为与领养人身份的绑定关系。
### 2.2 项目需求
本项目已完成流浪动物医疗筹款救助模块，为了形成「救治筹款-康复领养-全链路溯源」的完整公益闭环，需要开发独立的宠物领养模块，利用区块链不可篡改、永久留痕的特性，从根源上解决现有领养场景的核心痛点，为流浪动物领养提供可信的落地方案。

## 三、需求目标
### 3.1 核心目标
1.  实现宠物领养全流程的链上存证，所有操作记录永久留存、不可篡改，实现领养行为的全链路可追溯；
2.  实现领养人实名信息的合规链上存证，在保护用户隐私的前提下，解决领养人抵赖、信息灭失的核心问题；
3.  打造完整的领养业务闭环，覆盖「发布领养信息-提交领养申请-审核-领养完成」的全流程。
### 3.2 次要目标
1.  模块与现有筹款模块完全解耦，可独立运行，也可无缝对接现有项目架构；
2.  操作流程完全贴合线下领养的真实场景，降低用户学习成本，提升可用性；
3.  模块具备通用性，可单独复用给各类流浪动物救助机构、领养平台。

## 六、详细功能说明
### 6.1 领养信息管理模块
#### 6.1.1 发布宠物领养信息
- **前置条件**：用户已连接钱包，完成地址登录
- **操作流程**：
  1. 用户进入领养信息发布页面，填写宠物基础信息：宠物名称、品种、年龄、情况描述、宠物照片链接；
  2. 用户点击【发布】按钮，触发链上合约调用；
  3. 钱包确认交易后，信息写入链上，生成唯一的宠物ID；
  4. 发布成功，页面跳转到宠物领养详情页。
- **后置条件**：
  1. 链上生成新的宠物领养信息记录，状态为「待领养」；
  2. 宠物信息加入全平台领养信息列表，可被所有用户查看。
- **业务规则**：
  1. 宠物名称、品种、年龄为必填项，不能为空；
  2. 宠物年龄以月为单位，支持0及以上的正整数；
  3. 发布成功后，宠物基础信息不可修改，仅可更新领养状态；
  4. 只有发布该宠物的救助人，可管理该宠物的领养申请。

#### 6.1.2 查看领养信息列表
- **前置条件**：无，所有用户均可查看
- **操作流程**：
  1. 用户进入领养列表页面，系统自动从链上拉取所有宠物领养信息；
  2. 列表按发布时间倒序排列，最新发布的在前；
  3. 用户可查看宠物的基础信息：照片、名称、品种、年龄、领养状态；
  4. 用户可点击单条宠物信息，进入详情页。
- **业务规则**：
  1. 列表默认展示所有状态的宠物，支持筛选「待领养」「已领养」状态；
  2. 已领养的宠物，不可提交新的领养申请。

#### 6.1.3 查看领养信息详情
- **前置条件**：无，所有用户均可查看
- **操作流程**：
  1. 用户从领养列表点击进入宠物详情页；
  2. 页面展示宠物的完整基础信息、发布人地址、发布时间、申请总人数、领养状态；
  3. 救助人可查看该宠物的所有领养申请入口；
  4. 领养人可查看【提交领养申请】按钮。
- **业务规则**：
  1. 只有发布该宠物的救助人，可查看对应的领养申请列表；
  2. 已领养的宠物，隐藏【提交领养申请】按钮。

### 6.2 领养人实名存证模块
#### 6.2.1 实名信息登记
- **前置条件**：用户已连接钱包，完成地址登录，未完成过实名登记
- **操作流程**：
  1. 用户进入实名登记页面，按照提示，将姓名、身份证号、手机号分别通过SHA256加密工具生成对应的哈希指纹；
  2. 用户将3个哈希指纹填写到对应输入框，点击【提交登记】按钮，触发链上合约调用；
  3. 钱包确认交易后，哈希指纹写入链上，与用户的钱包地址永久绑定；
  4. 登记成功，页面提示实名登记完成。
- **后置条件**：
  1. 链上生成该用户的实名存证记录，状态为「已实名」；
  2. 用户获得提交领养申请的权限。
- **业务规则**：
  1. 一个钱包地址仅可完成一次实名登记，登记成功后不可修改、不可删除；
  2. 仅登记哈希指纹，不存储任何明文敏感信息，完全保护用户隐私；
  3. 登记时间永久写入链上，不可篡改。

#### 6.2.2 实名信息验证
- **前置条件**：用户已连接钱包，为对应宠物的发布人/平台管理员
- **操作流程**：
  1. 用户进入领养申请详情页，点击【验证实名信息】按钮；
  2. 用户输入待验证的姓名、身份证号、手机号的哈希指纹，点击【开始验证】；
  3. 系统调用链上合约，对比输入的哈希指纹与链上存储的哈希指纹是否完全一致；
  4. 页面返回验证结果：匹配/不匹配。
- **业务规则**：
  1. 仅当3项哈希指纹完全匹配时，才返回验证通过；
  2. 任意一项信息修改，都会导致哈希指纹不匹配，无法通过验证；
  3. 验证操作不会泄露任何明文信息，仅做哈希对比。

### 6.3 领养申请管理模块
#### 6.3.1 提交领养申请
- **前置条件**：用户已连接钱包，完成实名登记，对应宠物为「待领养」状态
- **操作流程**：
  1. 用户进入宠物领养详情页，点击【提交领养申请】按钮；
  2. 用户填写领养申请留言、领养承诺，点击【提交申请】，触发链上合约调用；
  3. 钱包确认交易后，申请信息写入链上，生成唯一的申请ID；
  4. 提交成功，页面提示申请提交完成。
- **后置条件**：
  1. 链上生成新的领养申请记录，状态为「待审核」；
  2. 申请记录加入对应宠物的申请列表，可被发布人查看；
  3. 对应宠物的申请总人数+1。
- **业务规则**：
  1. 未完成实名登记的用户，无法提交领养申请，按钮置灰，提示先完成实名登记；
  2. 一个用户针对同一只宠物，仅可提交一次领养申请；
  3. 已领养的宠物，无法提交领养申请；
  4. 申请提交成功后，申请留言不可修改。

#### 6.3.2 查看领养申请列表
- **前置条件**：用户已连接钱包，为对应宠物的发布人
- **操作流程**：
  1. 用户进入宠物领养详情页，点击【查看申请列表】按钮；
  2. 页面拉取该宠物对应的所有领养申请，按提交时间倒序排列；
  3. 列表展示申请ID、领养人地址、提交时间、申请状态；
  4. 用户可点击单条申请，进入详情页。
- **业务规则**：
  1. 仅宠物的发布人可查看该宠物的申请列表，其他用户无权限查看；
  2. 列表展示所有状态的申请，支持按状态筛选。

#### 6.3.3 审核领养申请
- **前置条件**：用户已连接钱包，为对应宠物的发布人，申请状态为「待审核」
- **操作流程**：
  1. 用户进入领养申请详情页，查看申请信息、领养人实名状态；
  2. 用户可选择【通过】或【拒绝】按钮，触发链上合约调用；
  3. 钱包确认交易后，审核结果写入链上，更新申请状态；
  4. 审核完成，页面提示操作成功。
- **后置条件**：
  1. 若审核通过：申请状态更新为「审核通过」，对应宠物的状态更新为「已领养」，该宠物的其他所有申请自动失效，不可再审核；
  2. 若审核拒绝：申请状态更新为「审核拒绝」，宠物状态保持「待领养」，可继续审核其他申请。
- **业务规则**：
  1. 仅宠物的发布人可审核对应申请，其他用户无权限操作；
  2. 一份申请仅可审核一次，审核完成后不可修改结果；
  3. 同一只宠物仅可通过一份申请，通过后宠物自动标记为已领养，不可再提交新的申请。

### 6.4 数据查询模块
#### 6.4.1 宠物申请列表查询
- 功能说明：所有用户可通过宠物ID，查询该宠物对应的所有领养申请编号列表
- 前置条件：宠物ID存在
- 业务规则：仅返回申请编号，不返回申请详情，详情仅发布人可查看

#### 6.4.2 领养人实名状态查询
- 功能说明：所有用户可通过钱包地址，查询该地址是否完成实名登记
- 前置条件：无
- 业务规则：仅返回是否实名的状态，不返回实名的哈希详情，保护用户隐私

## 七、核心业务流程
1.  **救助人发布领养信息流程**：连接钱包→填写宠物信息→提交发布→链上生成领养信息→进入领养列表
2.  **领养人领养全流程**：连接钱包→完成实名哈希登记→查看领养列表→进入宠物详情→提交领养申请→等待救助人审核→审核通过/拒绝
3.  **救助人审核流程**：查看自己发布的宠物→进入申请列表→查看申请详情→审核通过/拒绝→更新申请与宠物状态
4.  **实名验证流程**：获取领养人原文信息→生成哈希指纹→调用链上验证接口→返回验证结果→确认领养人身份

## 九、非功能需求
### 9.1 安全需求
- 合约代码遵循Solidity安全开发规范，规避重入、整数溢出、越权操作等常见安全漏洞；
- 所有敏感操作均设置严格的权限校验，仅对应角色可操作，杜绝越权风险；
- 所有链上数据均不可篡改、不可删除，保障存证的可信度。
### 9.2 兼容需求
- 合约采用0.8.20编译器版本，与项目现有筹款模块完全兼容，可无缝对接；
- 支持主流钱包（MetaMask、Coinbase Wallet等）的连接与调用；
- 可部署至Avalanche主网与测试网，兼容EVM链的所有标准。
### 9.3 合规需求
- 仅存储实名信息的哈希指纹，不存储任何明文敏感信息，完全符合《个人信息保护法》的合规要求；
- 不收集、不存储用户的明文隐私信息，所有隐私数据由用户自行保管。
### 9.4 性能需求
- 合约所有函数调用均为单链上操作，无复杂循环，gas消耗控制在合理范围内；
- 前端数据查询响应时间不超过2秒，保证用户操作的流畅性。

## 十、验收标准
### 10.1 功能验收标准
